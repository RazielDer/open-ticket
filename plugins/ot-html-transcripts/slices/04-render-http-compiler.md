# Slice 04: Render, HTTP Server, and Compiler Replacement

- Phase: `P04`
- Status: `ready`
- Depends on: `03-collection-dto-assets`
- Allowed writes: `plugins/ot-html-transcripts/**` only

## Objective

Render self-contained transcript archives, serve them locally, and replace the upstream html compiler without changing the core ready/failure UX.

## Deliverables

- `index.html` renderer
- public transcript and asset HTTP routes
- compiler override for `opendiscord:html-compiler`

## Target file plan

- `render/html-renderer.ts`

## Implementation tasks

1. Implement the archive build flow in the exact order:
   - create DB row with `status=building`
   - create temp archive directory
   - collect full history
   - build `document.json`
   - mirror assets
   - render `index.html`
   - atomically rename temp directory into final archive path
   - update DB row to `active` or `partial`
2. Render `index.html` as a self-contained page:
   - inline CSS
   - minimal local JS only if required
   - no CDN assets
   - enough local assets that old transcripts remain viewable on their own
3. Escape and sanitize all rendered Discord content and URLs before writing HTML.
4. Start a plugin-owned `node:http` server on `onReadyForUsage` using plugin config values.
5. Implement routes:
   - `GET /health`
   - `GET /transcripts/:slug`
   - `GET /transcripts/:slug/assets/:assetName`
6. Route behavior:
   - active slug returns transcript HTML
   - unknown slug returns `404`
   - revoked, superseded, or deleted slug returns `410`
   - asset route serves only assets belonging to the transcript resolved by the slug
7. HTTP headers:
   - transcript HTML: `Cache-Control: no-store`
   - mirrored assets: long immutable cache
   - `X-Content-Type-Options: nosniff`
   - strict CSP forbidding remote script execution
8. Defend against path traversal on every file-serving route.
9. Implement compiler replacement in `afterTranscriptCompilersLoaded`:
   - overwrite `opendiscord:html-compiler` with `overwrite:true`
   - keep compile result data shape `{ url:string, availableUntil:Date }`
   - use a far-future timestamp for compatibility
10. Keep the rest of the ready path unchanged:
   - do not edit core ready-message builders
   - do not edit `opendiscord:transcript-html-visit`
   - ensure `result.data.url` is the local transcript URL

## Integration behavior

- The compile path should delegate to the service and archive builder rather than reimplement queue or DB logic.
- `/health` should expose `isHealthy`, summary counters, queue depth, and recovered build count.
- Compiler failure must still flow back through the existing core transcript error message.

## Exit criteria

- A built transcript can be opened from the generated local URL.
- The visit button generated by existing core builders points at a working local page.
- Unknown, revoked, superseded, and deleted slugs return the correct status codes.
- No external transcript API call occurs when html mode is active with the plugin loaded.
